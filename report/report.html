1.Εισαγωγή και Arbitrage

1.1 Τι είναι το arbitrage στις στοιχηματικές αγορές

Σε μια στοιχηματική αγορά με τρία πιθανά αποτελέσματα (Νίκη Γηπεδούχου, Ισοπαλία, Νίκη Φιλοξενούμενου — γνωστή ως 1Χ2), κάθε στοιχηματική εταιρία προσφέρει τη δική της δεκαδική απόδοση για κάθε outcome. Η αντίστροφη τιμή 1/odds αντιστοιχεί στην εξυπακουόμενη πιθανότητα (implied probability). Σε μια ιδεατή «δίκαιη» αγορά χωρίς περιθώριο κέρδους για τον bookmaker, το άθροισμα των τριών εξυπακουόμενων πιθανοτήτων ισούται με 1. Στην πραγματικότητα είναι μεγαλύτερο, καθώς κάθε στοιχηματική εισάγει overround.

Όταν όμως ληφθούν οι καλύτερες αποδόσεις κάθε outcome από διαφορετικές στοιχηματικές, το άθροισμα μπορεί ενίοτε να βρεθεί μικρότερο της μονάδας:

arbitrage ⟺  1/best_odds_1 + 1/best_odds_X + 1/best_odds_2  <  1.0
profit_margin  =  1 − Σ(1/best_odds_i)

Σε αυτή την περίπτωση, ο παίκτης μπορεί να στοιχηματίσει σε όλα τα outcomes με κατάλληλα κατανεμημένα ποσά (αναλογικά με το 1/odds) και να εξασφαλίσει εγγυημένο κέρδος ίσο με το profit_margin, ανεξάρτητα από το τελικό αποτέλεσμα του αγώνα.

Παράδειγμα

Έστω καλύτερες αποδόσεις 2.20 / 3.60 / 4.00 από τρεις διαφορετικές στοιχηματικές. Το άθροισμα 1/2.20 + 1/3.60 + 1/4.00 = 0.455 + 0.278 + 0.250 = 0.983 < 1.0. Υπάρχει arbitrage με περιθώριο 1.7%. Στοιχηματίζοντας ποσά αναλογικά (45.5% του συνόλου στο 1, 27.8% στο Χ, 25.0% στο 2), η επιστροφή είναι ίδια σε κάθε ένα από τα τρία πιθανά αποτελέσματα και υπερβαίνει το αρχικό κεφάλαιο κατά 1.7%.

1.2 Γιατί το πρόβλημα είναι κατεξοχήν streaming

Οι αποδόσεις στις στοιχηματικές μεταβάλλονται συνεχώς λόγω εισροής στοιχημάτων από το κοινό, εσωτερικών μοντέλων αξιολόγησης κινδύνου, ή νέων πληροφοριών για τον αγώνα (τραυματισμοί, καιρικές συνθήκες, κ.λπ.). Μια arbitrage ευκαιρία δημιουργείται από στιγμιαία ασυμμετρία μεταξύ δύο ή περισσότερων bookmakers και τυπικά διαρκεί μερικά δευτερόλεπτα πριν οι αυτόματοι αλγόριθμοι των στοιχηματικών την εξαλείψουν.

Συνεπώς, οποιοδήποτε σύστημα προσπαθεί να εντοπίσει τέτοιες ευκαιρίες πρέπει να εμφανίζει χαμηλή καθυστέρηση από τη στιγμή που μια απόδοση αλλάζει μέχρι τη στιγμή που εκπέμπεται μια ειδοποίηση arbitrage. Αυτό καθιστά το πρόβλημα ιδανικό για παραδειγματική σύγκριση μεταξύ δύο διαφορετικών προσεγγίσεων κατανεμημένης επεξεργασίας: Batch (περιοδική επεξεργασία συσσωρευμένων δεδομένων) έναντι Stream (συνεχής επεξεργασία κάθε event μόλις φτάνει).

1.3 Στόχοι της εργασίας

  • Ανάπτυξη streaming pipeline βασισμένου σε Apache Kafka και Apache Spark.
  • Υλοποίηση δύο ξεχωριστών consumers (Batch και Stream) με κοινή λογική ανίχνευσης arbitrage.
  • Πειραματική σύγκριση των δύο μεθόδων σε τέσσερις διαστάσεις: χρονική καθυστέρηση εντοπισμού (detection latency), διεκπεραιωτική ικανότητα (throughput), σταθερότητα χρόνου επεξεργασίας, και πληρότητα ανίχνευσης (catch rate).
  • Παραγωγή ποσοτικών αποτελεσμάτων που επιτρέπουν τεκμηριωμένη επιλογή αρχιτεκτονικής για παρόμοια προβλήματα.

2.Θεωρητικό υπόβαθρο

2.1 Big Data και τα τέσσερα V

Ως Big Data ορίζεται σύνολο δεδομένων το οποίο, λόγω όγκου ή πολυπλοκότητας, δεν μπορεί να επεξεργαστεί με παραδοσιακό λογισμικό. Χαρακτηρίζεται από τα τέσσερα V: Volume (όγκος), Velocity (ταχύτητα παραγωγής), Variety (ποικιλία μορφών) και Veracity (αξιοπιστία). Στην παρούσα εργασία κυριαρχεί το χαρακτηριστικό Velocity: τα δεδομένα παράγονται σε υψηλό ρυθμό και απαιτούν άμεση επεξεργασία.

2.2 Batch vs Stream Processing

Batch processing: τα δεδομένα συγκεντρώνονται σε χρονικά παράθυρα προκαθορισμένης διάρκειας και επεξεργάζονται μαζικά μετά την ολοκλήρωση του παραθύρου. Παρέχει υψηλή απόδοση σε σταθερό φόρτο εργασίας αλλά εισάγει εγγενή καθυστέρηση τουλάχιστον ίση με το παράθυρο.

Stream processing: τα events επεξεργάζονται μεμονωμένα ή σε πολύ μικρά micro-batches μόλις φτάσουν. Εξασφαλίζει χαμηλή καθυστέρηση αντίδρασης αλλά απαιτεί διαχείριση κατάστασης (state) μεταξύ διαδοχικών events.

2.3 Apache Kafka

Το Apache Kafka είναι κατανεμημένο σύστημα μηνυμάτων (distributed message broker) σχεδιασμένο για υψηλό throughput και ανθεκτικότητα. Λειτουργεί ως ενδιάμεσος μεταξύ producers (που στέλνουν μηνύματα) και consumers (που τα διαβάζουν), αποθηκεύοντάς τα σε topics τα οποία υποδιαιρούνται σε partitions για παραλληλισμό. Από την έκδοση 3.x υποστηρίζει τη λειτουργία KRaft, η οποία αντικαθιστά το Zookeeper με ενσωματωμένο μηχανισμό consensus.

2.4 Apache Spark

Το Apache Spark είναι framework κατανεμημένης επεξεργασίας δεδομένων που, σε σύγκριση με τον προγενέστερο Hadoop MapReduce, παρέχει σημαντικά βελτιωμένη απόδοση (συνήθως 10× έως 100× σε τυπικά benchmarks) και ευκολότερο API. Υποστηρίζει τόσο batch επεξεργασία (Spark SQL / DataFrame API) όσο και streaming (Spark Structured Streaming) μέσα από κοινή προγραμματιστική διεπαφή, γεγονός που επιτρέπει «apples-to-apples» σύγκριση μεταξύ των δύο παραδειγμάτων.

2.5 Λογική ανίχνευσης Arbitrage

Σε στιγμιότυπο μιας αγοράς 1Χ2 με Ν διαφορετικές στοιχηματικές, η ανίχνευση arbitrage συνοψίζεται σε τρία βήματα:

  1. Για κάθε outcome (H/X/A), εντοπίζεται η μέγιστη διαθέσιμη δεκαδική απόδοση μεταξύ όλων των N στοιχηματικών.
  2. Υπολογίζεται το άθροισμα των αντιστρόφων αυτών των τριών μέγιστων αποδόσεων.
  3. Εάν το άθροισμα είναι μικρότερο της μονάδας (με αμελητέο ε για αποφυγή float noise), έχει εντοπιστεί arbitrage. Το profit margin ισούται με 1 − άθροισμα.

3.Αρχιτεκτονική

Η αρχιτεκτονική ακολουθεί κλασικό streaming pattern τριών επιπέδων: παραγωγός δεδομένων → broker → καταναλωτές. Όλα τα στοιχεία τρέχουν τοπικά (η Θεματική 1 δεν απαιτεί cloud provider): το Apache Kafka σε container Docker, οι Spark consumers σε λειτουργία local[*] που αξιοποιεί όλους τους διαθέσιμους πυρήνες του μηχανήματος.

4.Δεδομένα

4.1 Πηγή

Τα δεδομένα προέρχονται από το football-data.co.uk, μια δημόσια αρχειακή πηγή που δημοσιεύει αποτελέσματα ποδοσφαιρικών αγώνων και αντίστοιχες αποδόσεις πολλαπλών στοιχηματικών. Η επιλογή αυτή προτιμήθηκε λόγω άμεσης διαθεσιμότητας των CSV χωρίς λογαριασμό ή έλεγχο ταυτότητας.

Επελέγησαν 8 ευρωπαϊκές λίγκες × 5 σεζόν (2020–21 έως 2024–25), συνολικά 40 αρχεία CSV και 14.802 αγώνες, καλύπτοντας: Αγγλικό Premier League, Αγγλικό Championship, Ισπανικό La Liga, Ιταλικό Serie A, Γερμανικό Bundesliga, Γαλλικό Ligue 1, Ολλανδικό Eredivisie και Πορτογαλικό Primeira Liga.

Σημαντικός περιορισμός της πηγής

Το dataset παρέχει αποδόσεις τύπου closing/opening ανά αγώνα — όχι intra-match time series. Συνεπώς το arbitrage εντοπίζεται μέσω cross-bookmaker σύγκρισης για ένα δεδομένο γεγονός σε συγκεκριμένο στιγμιότυπο, και όχι μέσω παρακολούθησης της εξέλιξης των αποδόσεων κατά τη διάρκεια του παιχνιδιού.

4.3 Προκαταρκτική σάρωση arbitrage

Πριν την υλοποίηση του streaming pipeline, εκτελέστηκε offline σάρωση όλου του dataset για επιβεβαίωση ύπαρξης ευκαιριών arbitrage. Στο σύνολο των 14.798 αγώνων με τουλάχιστον δύο διαθέσιμες στοιχηματικές, εντοπίστηκαν 90 ευκαιρίες, δηλαδή ποσοστό 0,608%. Το αποτέλεσμα ευθυγραμμίζεται με τη βιβλιογραφία, η οποία αναφέρει τυπικά ποσοστά 0,5%–1% για pre-match arbitrage σε ποδόσφαιρο. Το μεγαλύτερο εντοπισμένο περιθώριο ήταν 9,34% στον αγώνα Paris SG–Metz (16/9/2020).

5.2 Producer

Ο producer (producer/producer.py) διαβάζει ολόκληρο το σύνολο των CSV, ταξινομεί τις εγγραφές χρονολογικά με βάση τα πεδία Date και Time, και για κάθε αγώνα παράγει ένα event ανά στοιχηματική που διαθέτει έγκυρες αποδόσεις 1Χ2. Κάθε event έχει την ακόλουθη δομή JSON:

{
  "match_id":      "F1-2020-08-21-Bordeaux-Nantes",
  "league":        "F1",
  "home_team":     "Bordeaux",
  "away_team":     "Nantes",
  "match_date":    "2020-08-21",
  "bookmaker":     "Bet365",
  "odds_home":     2.4,
  "odds_draw":     3.0,
  "odds_away":     3.25,
  "event_time_ms": 1598032800000,
  "real_time_ms":  1779490227596
}

Το event περιλαμβάνει δύο χρονοσφραγίδες με διακριτό ρόλο: το event_time_ms αντιστοιχεί στη χρονική στιγμή τέλεσης του αγώνα (χρησιμοποιείται για χρονική ομαδοποίηση downstream), ενώ το real_time_ms καταγράφει την wall-clock στιγμή αποστολής του event στο Kafka (χρησιμοποιείται για τον υπολογισμό του end-to-end latency στους consumers).

5.3 Μέθοδος Α — Batch Consumer

Ο Batch consumer (consumers/batch/batch_consumer.py) ακολουθεί λογική polling loop. Σε κάθε επανάληψη του βρόχου:

  1. Ζητούνται από το Kafka τα τρέχοντα end offsets για κάθε partition του topic μέσω σύντομου κύκλου ζωής kafka-python client.
  2. Εκτελείται Spark batch read με ρητά JSON offsets ορίζοντας ως starting offsets τα προηγούμενα ending offsets και ως ending offsets τα τρέχοντα.
  3. Με χρήση window function διατηρείται μόνο η πιο πρόσφατη απόδοση ανά συνδυασμό (match_id, bookmaker).
  4. Για κάθε αγώνα με τουλάχιστον δύο στοιχηματικές, καλείται η detect_arbitrage().
  5. Ο βρόχος αναπαύεται για χρόνο ίσο με το window_seconds πριν την επόμενη επανάληψη.

Η αναμενόμενη μέση καθυστέρηση εντοπισμού είναι περίπου ίση με το ήμισυ του παραθύρου, ενώ η μέγιστη φτάνει το πλήρες παράθυρο.

5.4 Μέθοδος Β — Stream Consumer

Ο Stream consumer (consumers/stream/stream_consumer.py) αξιοποιεί το Spark Structured Streaming με trigger processingTime ορισμένο στα 200 ms. Σε κάθε micro-batch, ένα foreachBatch callback εκτελεί τα ακόλουθα βήματα:

  1. Ενημερώνεται ένα Python dictionary που διατηρεί τις τελευταίες αποδόσεις ανά (match_id, bookmaker).
  2. Καταγράφονται τα touched matches, δηλαδή οι αγώνες που έλαβαν τουλάχιστον μία ενημέρωση στο τρέχον micro-batch.
  3. Η detect_arbitrage() καλείται μόνο για τα touched matches (incremental work), όχι για όλους τους γνωστούς αγώνες.
  4. Εφαρμόζεται de-duplication μέσω έλεγχου μεταβολής του profit margin: μια ευκαιρία δεν επανεκπέμπεται αν το νέο margin διαφέρει λιγότερο από 10⁻⁴ από το προηγουμένως αναφερθέν.

5.5 Λογική ανίχνευσης

Η arbitrage/detector.py υλοποιεί τη μαθηματική λογική που περιγράφηκε στην παράγραφο 2.5. Αποτελεί τη μοναδική πηγή αλήθειας για την ανίχνευση και χρησιμοποιείται από αμφότερους τους consumers, εξασφαλίζοντας έτσι ότι τυχόν διαφορές αποτελεσμάτων μεταξύ Batch και Stream οφείλονται αποκλειστικά στην αρχιτεκτονική επιλογή και όχι σε διαφορετική λογική υπολογισμού.

5.6 Σύγκριση κώδικα: πού ακριβώς διαφέρουν οι δύο υλοποιήσεις

Παρότι οι δύο consumers μοιράζονται την ίδια λογική ανίχνευσης, ο τρόπος με τον οποίο καταναλώνουν τα δεδομένα από το Kafka είναι θεμελιωδώς διαφορετικός. Η πρώτη διαφορά εντοπίζεται στο σημείο ανάγνωσης: ο Batch χρησιμοποιεί spark.read (πεπερασμένη ανάγνωση με ρητά όρια offsets), ενώ ο Stream χρησιμοποιεί spark.readStream (συνεχές query χωρίς «τέλος»).

Batch — spark.read (batch_consumer.py)
# Polling loop: ο ΔΙΚΟΣ μας κώδικας αποφασίζει
# πότε και τι θα διαβαστεί.
while not stop:
    # 1. Ρωτάμε το Kafka πού βρίσκεται το tail
    end_offsets = fetch_end_offsets(servers, topic)

    # 2. Πεπερασμένο read: από τον cursor
    #    μέχρι τα τρέχοντα end offsets
    raw = (spark.read.format("kafka")
        .option("subscribe", topic)
        .option("startingOffsets",
                starting_offsets_json)
        .option("endingOffsets",
                ending_offsets_json)
        .load())

    # ... detection σε ΟΛΟ το παράθυρο ...

    # 3. Χειροκίνητη προώθηση cursor
    cursor = dict(end_offsets)

    # 4. Αδράνεια μέχρι το επόμενο παράθυρο
    time.sleep(window_seconds - elapsed)
Stream — spark.readStream (stream_consumer.py)
# Συνεχές streaming query: το SPARK αποφασίζει
# πότε τρέχει το επόμενο micro-batch.
raw = (spark.readStream.format("kafka")
    .option("subscribe", topic)
    .option("startingOffsets", "earliest")
    .load())

# Δηλωτικός ορισμός: callback ανά micro-batch
query = (events.writeStream
    .foreachBatch(state.process_batch)
    # Τα offsets τα διαχειρίζεται ΤΟ SPARK
    # αυτόματα μέσω checkpoint directory
    .option("checkpointLocation",
            str(checkpoint_dir))
    # Νέο micro-batch κάθε 200 ms
    .trigger(processingTime="200 milliseconds")
    .start())

# Ο driver απλώς περιμένει — δεν υπάρχει
# polling loop ούτε sleep()
query.awaitTermination()

Η δεύτερη διαφορά αφορά τη διατήρηση του «τελευταίου» odds ανά (αγώνα, στοιχηματική). Ο Batch, επειδή βλέπει κάθε φορά ένα αυτοτελές σύνολο δεδομένων, το λύνει δηλωτικά με SQL window function. Ο Stream, επειδή βλέπει μόνο τα νέα events κάθε micro-batch, χρειάζεται state που επιβιώνει μεταξύ διαδοχικών micro-batches:

Batch — stateless, SQL window function
# Κάθε batch είναι αυτοτελές: η deduplication
# γίνεται δηλωτικά μέσα στο DataFrame.
w = F.expr(
    "row_number() over ("
    "  partition by match_id, bookmaker"
    "  order by event_time_ms desc)")

snapshot = (df.withColumn("_rn", w)
              .filter("_rn = 1")
              .drop("_rn"))

# Detection σε ΟΛΑ τα matches του batch
opps = detect_for_partition(
    snapshot.collect())
Stream — stateful, incremental ενημέρωση
# Το state ζει ΕΞΩ από το micro-batch και
# ενημερώνεται incremental σε κάθε callback.
cur = self.latest.setdefault(mid, {}).get(book)
if cur is None or ev[0] >= cur[0]:
    self.latest[mid][book] = ev
    touched_matches.add(mid)

# Detection ΜΟΝΟ στα matches που άλλαξαν
# σε αυτό το micro-batch (incremental work)
for mid in touched_matches:
    opp = detect_arbitrage(
        match_id=mid,
        home_odds=..., draw_odds=...,
        away_odds=...,
        detected_at_ms=now_ms)

6.Πειραματική μεθοδολογία

6.1 Πειραματικό περιβάλλον

Όλα τα πειράματα εκτελέστηκαν σε MacBook με Apple Silicon. Το Kafka τρέχει σε container Docker με 1 broker και 3 partitions· το Spark σε λειτουργία local[*] που αξιοποιεί όλους τους πυρήνες του μηχανήματος. Δεν χρησιμοποιείται cluster πολλαπλών κόμβων· η σύγκριση παραδειγμάτων είναι ανεξάρτητη από αυτό, καθώς οι μετρήσεις αφορούν την αρχιτεκτονική επιλογή και όχι την κλιμακωσιμότητα.

6.2 Αυτοματοποιημένος orchestrator

Για διασφάλιση της επαναληψιμότητας, αναπτύχθηκε orchestrator (benchmarks/run_experiment.py) που αυτοματοποιεί τα παρακάτω βήματα ανά πείραμα:

  1. Διαγραφή και αναδημιουργία του Kafka topic (καθαρή κατάσταση).
  2. Εκκίνηση των δύο consumers σε ξεχωριστές υποδιεργασίες.
  3. Αναμονή 5 δευτερολέπτων warmup για ολοκλήρωση της εκκίνησης του Spark JVM.
  4. Εκτέλεση του producer.
  5. Drain period μετά την ολοκλήρωση του producer, ώστε οι consumers να επεξεργαστούν το backlog.
  6. Αποστολή SIGINT στους consumers για graceful shutdown.
  7. Αποθήκευση των αποτελεσμάτων σε tagged υποφάκελο μαζί με αρχείο config.json που περιλαμβάνει όλα τα CLI args.

7.3 Σύγκριση latency

Κεντρικό εύρημα

Όταν το batch window τριπλασιάζεται από 10 s σε 30 s, η median Batch latency τριπλασιάζεται αντίστοιχα (9,75 s → 29,87 s). Η median Stream latency παραμένει πρακτικά σταθερή (1,43 s → 1,57 s). Το εύρημα επιβεβαιώνει τη θεωρητική σχέση batch latency ≈ window, η οποία προκύπτει από το γεγονός ότι ένα event το οποίο φτάνει στο Kafka αμέσως μετά την έναρξη ενός batch αναμένει σχεδόν ολόκληρο το παράθυρο μέχρι την επόμενη εκτέλεση.

7.5 Throughput

Το αποτέλεσμα αυτό αντιβαίνει στη συνήθη παραδοχή ότι το batch processing παρέχει μεγαλύτερη διεκπεραιωτική ικανότητα. Η εξήγηση εντοπίζεται στο γεγονός ότι ο Batch consumer δαπανά σημαντικό μέρος του χρόνου του σε idle waits μεταξύ διαδοχικών batches (sleep μέχρι τη συμπλήρωση του παραθύρου), ενώ ο Stream consumer επεξεργάζεται συνεχώς. Σε εργασίες με σταθερό φόρτο και μεγαλύτερα παράθυρα, η σύγκριση ενδέχεται να αντιστραφεί.

7.6 Σταθερότητα χρόνου επεξεργασίας

Η median per-batch processing time είναι παραπλήσια στις δύο μεθόδους (345 ms Batch, 541 ms Stream), αλλά ο Batch consumer εμφανίζει σημαντικά μεγαλύτερη διασπορά: το p95 ξεπερνά τα 3,2 s, ενώ του Stream παραμένει κάτω από 800 ms. Αυτό οφείλεται στο ότι το πρώτο batch κάθε εκτέλεσης πρέπει να καταναλώσει το πλήρες backlog ταυτόχρονα, ενώ τα stream micro-batches έχουν σταθερά μικρά μεγέθη.

8.Επίδειξη εκτέλεσης

Η ενότητα αυτή τεκμηριώνει βήμα-βήμα μια πλήρη εκτέλεση του pipeline, όπως παρουσιάστηκε και στη ζωντανή επίδειξη της εργασίας. 8.1 Βήμα 1 — Εκκίνηση υποδομής και καθαρό topic

Το Kafka (KRaft mode) και το Kafka UI εκκινούν ως Docker containers με docker compose up -d. Στη συνέχεια το topic betting-odds διαγράφεται και αναδημιουργείται με το εργαλείο kafka-topics.sh (εκτελείται μέσα στο container μέσω docker exec) με 3 partitions, ώστε κάθε εκτέλεση να ξεκινά από μηδενική κατάσταση — χωρίς παλιά μηνύματα, offsets ή checkpoints.

8.2 Βήμα 2 — Εκκίνηση των δύο consumers

Σε δύο ξεχωριστά terminals εκκινούν ο Stream consumer (trigger 200 ms) και ο Batch consumer (window 8 s). Και οι δύο συνδέονται στο ίδιο topic και περιμένουν δεδομένα — ο Stream μέσω συνεχούς streaming query, ο Batch μέσω polling loop που «ξυπνά» κάθε 8 δευτερόλεπτα (τα κενά batches 1–4 στο Σχήμα 17 είναι ακριβώς αυτά τα άδεια παράθυρα πριν την άφιξη δεδομένων).

8.3 Βήμα 3 — Αποστολή 5.000 events

Ο producer διαβάζει και τα 14.802 matches, τα ταξινομεί χρονολογικά και στέλνει τα πρώτα 5.000 events στο Kafka με μέγιστη ταχύτητα (--speed 0). Η αποστολή ολοκληρώνεται σε 0,3 δευτερόλεπτα με μέσο ρυθμό ~15.000 msg/s — απόδειξη ότι ο producer δεν αποτελεί bottleneck της σύγκρισης.

Η άφιξη των μηνυμάτων επιβεβαιώνεται στο Kafka UI: 5.000 μηνύματα κατανεμημένα στα 3 partitions (1.719 / 1.548 / 1.733). Η σχεδόν ομοιόμορφη κατανομή προκύπτει από το hashing του partition key (match_id).

8.4 Βήμα 4 — Οι δύο consumers εντοπίζουν τα arbitrage

Ο Stream consumer παραλαμβάνει και τα 5.000 events στο αμέσως επόμενο micro-batch: στο batch=2 επεξεργάζεται 5.000 rows που αφορούν 1.253 αγώνες (touched matches) και εντοπίζει και τις 7 ευκαιρίες σε 1.110 ms — περίπου 1,3 δευτερόλεπτα μετά την αποστολή.

Ο Batch consumer, αντίθετα, «κοιμάται» μέχρι τη λήξη του τρέχοντος παραθύρου: τα batches 1–4 είναι κενά, και τα δεδομένα επεξεργάζονται μαζικά στο batch=5 — περίπου 10 δευτερόλεπτα μετά την αποστολή (αναμονή παραθύρου + 3,5 s επεξεργασίας του πλήρους backlog). Η διαφορά αυτή είναι οπτικοποιημένη στιγμή προς στιγμή η θεωρητική σχέση batch latency ≈ window της ενότητας 7.

8.5 Βήμα 5 — Επαλήθευση αποτελεσμάτων

Κάθε consumer γράφει τις ευκαιρίες που εντοπίζει στο δικό του αρχείο εξόδου (stream_opps.jsonl / batch_opps.jsonl)· το demo.sh results απλώς τα διαβάζει και τα μορφοποιεί σε συγκριτική λίστα. Αμφότερες οι μέθοδοι εντοπίζουν τα ίδια 7 arbitrage — μεταξύ αυτών και το μεγαλύτερο περιθώριο του dataset (Paris SG–Metz, 9,34%) — επιβεβαιώνοντας ζωντανά την ταυτόσημη πληρότητα ανίχνευσης της παραγράφου 7.2: η διαφορά των δύο παραδειγμάτων είναι αμιγώς θέμα χρονισμού, όχι ορθότητας.

9.Συμπεράσματα

9.1 Σύνοψη ευρημάτων

  1. Ορθότητα. Αμφότερες οι μέθοδοι παράγουν ταυτόσημα σύνολα ανίχνευσης. Η επιλογή αρχιτεκτονικής δεν επηρεάζει την ακρίβεια του εντοπισμού.
  2. Latency. Ο Stream consumer υπερτερεί κατά παράγοντα 6,8× έως 19× ανάλογα με τη διαμόρφωση του Batch. Σημαντικότερο: η Stream latency είναι ανεξάρτητη του configuration, ενώ η Batch latency κλιμακώνεται γραμμικά με το παράθυρο.
  3. Throughput. Ο Stream consumer επιτυγχάνει 7–8× υψηλότερο throughput στο συγκεκριμένο πειραματικό setup, εύρημα που αντιβαίνει σε ευρέως διαδεδομένες αντιλήψεις περί υπεροχής του batch.
  4. Σταθερότητα. Η κατανομή per-batch processing time είναι σαφώς πιο συμπιεσμένη στη Stream μέθοδο.

9.2 Πρακτική σύσταση

Για ένα παραγωγικό σύστημα ανίχνευσης arbitrage σε στοιχηματικές αποδόσεις, η Stream αρχιτεκτονική είναι σαφώς προτιμότερη. Οι ευκαιρίες arbitrage έχουν διάρκεια ζωής της τάξης των δευτερολέπτων, καθιστώντας απαγορευτική την καθυστέρηση ενός batch consumer με window 30 s ή μεγαλύτερο. Η Batch αρχιτεκτονική παραμένει χρήσιμη μόνο για ιστορική ανάλυση και reporting όπου δεν απαιτείται απόκριση πραγματικού χρόνου.

9.3 Περιορισμοί της μελέτης

  • Single-machine setup. Η σύγκριση πραγματοποιήθηκε σε ένα μηχάνημα. Σε πραγματικό cluster πολλαπλών executors, οι σχετικές αναλογίες ενδέχεται να μεταβληθούν.
  • Pre-match odds μόνο. Το dataset δεν περιλαμβάνει intra-match (live) movements των αποδόσεων.
  • Driver-side state. Στον Stream consumer η κατάσταση διατηρείται σε Python dict στον driver, χωρίς δυνατότητα checkpointing ή recovery. Για παραγωγική κλίμακα 10⁷+ αγώνων θα απαιτούνταν stateful Spark operators με RocksDB state store.
  • Single-broker Kafka. Σε production setting θα χρησιμοποιούνταν cluster Kafka με replication factor τουλάχιστον 3.

9.4 Μελλοντικές επεκτάσεις

  1. Σύνδεση με live odds API (π.χ. The Odds API) για αξιολόγηση σε πραγματικό streaming φορτίο.
  2. Εκτέλεση σε multi-broker Kafka και multi-node Spark cluster για αξιολόγηση scalability.
  3. Αντικατάσταση του driver-side state με Spark mapGroupsWithState και RocksDB state store για υποστήριξη checkpointing και fault tolerance.
  4. Αξιολόγηση πιο επιθετικών configurations: trigger 50 ms στον Stream, μικρότερα windows στον Batch.
  5. Επέκταση σε επιπλέον αγορές (over/under goals, both teams to score), όπου η λογική arbitrage είναι αναλογικά εφαρμόσιμη.
